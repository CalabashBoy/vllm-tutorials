# vLLM 完全入门教程（一）：什么是 vLLM

## 1. 为什么需要 vLLM？

### 1.1 大模型推理的挑战

想象一下你在开发一个 AI 聊天应用：

```
用户A: "你好" → 服务器 → AI模型 → 回复
用户B: "你好" → 服务器 → AI模型 → 回复  
用户C: "你好" → 服务器 → AI模型 → 回复
```

**传统方式的问题：**
- 每个用户请求串行处理，排队等待
- GPU 内存利用率低（通常只有 20-40%）
- 响应慢，成本高

### 1.2 vLLM 能解决什么？

| 指标 | 传统推理 | vLLM | 提升 |
|------|---------|------|------|
| 吞吐量 | 100 req/s | 500+ req/s | **5倍+** |
| GPU 利用率 | 20-40% | 80-95% | **2-3倍** |
| 首 token 延迟 | 高 | 低 | **显著改善** |

**真实案例：**
- 某公司将 LLaMA-70B 部署从 8 张 A100 减少到 4 张
- 响应时间从 2 秒降到 0.5 秒
- 成本降低 60%

## 2. vLLM 的核心创新

### 2.1 PagedAttention（分页注意力）

这是 vLLM 最重要的创新，理解它是掌握 vLLM 的关键。

#### 类比：操作系统的虚拟内存

```
操作系统内存管理：
┌─────────────────┐
│ 程序A需要 4GB   │
│ 程序B需要 2GB   │
│ 程序C需要 8GB   │
│ 总共 16GB 内存   │
└─────────────────┘

实际做法：
- 不预分配连续空间
- 按需分配 4KB 页面
- 不用的页面换出到磁盘
```

**vLLM 把这套思想用到了 Attention 机制上！**

#### 传统 Attention 的问题

```python
# 传统方式：为每个请求预分配最大可能长度
request_memory = max_sequence_length × hidden_size × num_layers

# 例如：4096 长度 × 4096 维度 × 32 层 = 512MB/请求
# 10 个并发请求就需要 5GB，大部分空间浪费在等待生成的 token 上
```

**问题：** 请求刚进来时只发了 10 个 token，但要预留给 4096 个 token 的空间。

#### PagedAttention 的解决方案

```
把 KV Cache 分成固定大小的 "块"（Block），类似内存分页：

物理 GPU 内存：
┌─────┬─────┬─────┬─────┬─────┬─────┐
│ B0  │ B1  │ B2  │ B3  │ B4  │ B5  │  ...
└─────┴─────┴─────┴─────┴─────┴─────┘

请求A的逻辑视图（生成 "我爱中国"）：
┌─────────┬─────────┬─────────┐
│ 我→爱   │ 爱→中   │ 中→国   │
│ Block 0 │ Block 1 │ Block 2 │
└─────────┴─────┬───┴─────────┘
                │
                └── 物理上可能分散存储

请求B的逻辑视图（生成 "今天天气"）：
┌─────────┬─────────┐
│ 今天→天 │ 天→气   │
│ Block 3 │ Block 4 │
└─────────┴─────────┘
```

**关键优势：**
1. **按需分配**：需要多少分配多少，不浪费
2. **内存共享**：相同前缀的请求可以共享块
3. **灵活调度**：块可以分散存储，充分利用碎片空间

### 2.2 连续批处理 (Continuous Batching)

#### 传统批处理

```
时间轴：
0ms:  [ReqA========]  开始处理请求A（100 tokens）
100ms:     [ReqB========]  等A完成才能处理B
200ms:          [ReqC========]  等B完成才能处理C

问题：短请求要等长请求完成
```

#### 连续批处理

```
时间轴：
0ms:   [A=][B=][C=]      A,B,C 同时开始
20ms:  [A=][B=][D=]      A完成一步，C完成，新请求D进来
40ms:  [A=][B=][D=][E=]   动态调整批次

优势：
- GPU 一直在工作，没有空闲
- 新请求不用等
- 吞吐量大幅提升
```

### 2.3 张量并行 (Tensor Parallelism)

当模型太大（如 70B 参数），单张 GPU 放不下时：

```
单卡推理（装不下）：
┌─────────────────┐
│   70B 模型       │ 需要 140GB+ 显存
│   (1张 A100)     │ A100 只有 80GB
└─────────────────┘

张量并行（4卡）：
┌────────┬────────┬────────┬────────┐
│ GPU 0  │ GPU 1  │ GPU 2  │ GPU 3  │
│ 层1-8  │ 层9-16 │层17-24 │层25-32 │
│ 35GB   │ 35GB   │ 35GB   │ 35GB   │
└────────┴────────┴────────┴────────┘
        ↑ 并行计算，加速推理
```

| **性能** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| **社区活跃度** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **量化支持** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **多 GPU** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ |

**选择建议：**
- **新手入门/快速部署** → vLLM
- **极致性能/NVIDIA GPU** → TensorRT-LLM
- **HuggingFace 生态** → TGI
- **Mac/CPU/边缘设备** → llama.cpp

## 4. 安装前的准备

### 4.1 硬件要求

**最低配置（本地测试）：**
- GPU: NVIDIA GTX 1060 6GB 或更高
- 内存: 16GB+
- 硬盘: 50GB+（模型文件较大）

**推荐配置（生产环境）：**
- GPU: NVIDIA A100 / H100 / RTX 4090
- 显存: 24GB+
- 内存: 64GB+
- NVMe SSD: 100GB+

### 4.2 软件环境

```bash
# 检查 CUDA 版本（需要 CUDA 11.8+ 或 12.1+）
nvidia-smi

# 应该看到类似输出：
# NVIDIA-SMI 535.104.05   Driver Version: 535.104.05   CUDA Version: 12.2

# 检查 Python 版本（需要 Python 3.8-3.11）
python --version
```

### 4.3 网络准备

```bash
# 确保能访问 HuggingFace（国内需要代理或镜像）
curl -I https://huggingface.co
```

**关键数据：**
- 吞吐量提升 5 倍+
